## 1. 프로젝트 소개 (2분)

### 한 문장
> "법이 어려워서 망설이고 계신가요? 내 상황을 그대로 적으면 관련 법과 판례를 찾아드립니다."

### 문제 정의
- 일반인은 법률 문제가 생겨도 "어디서 시작해야 할지" 모름
- 변호사 자문은 비싸고 부담스러움
- 케이스노트, 엘박스 등은 변호사용 — 일반인이 보면 압도됨
- 검색해도 정확한 법령 조항을 찾기 어려움

### 우리 해결
- 자연어로 사연 입력 → 키워드 매칭 → 관련 법령/판례 자동 노출
- "법령 먼저, 판례 나중" — 1차 자료부터 보여줌
- 비슷한 사연 커뮤니티로 외로움 해소

### 비즈니스 모델
- **완전 무료** + 광고(AdSense) + 정부 지원금
- 변호사 매칭/광고 ❌ (변호사법 109조 회피선)
- 단정적 자문 ❌ (모든 응답에 면책 고지)

## 2. 데이터 구조 / 앱 구조 (2분)

### 5개 도메인 앱
```
accounts (사용자/JWT)
stories (사연 + 카테고리)        ← 도메인 핵심
legal_data (법령 + 판례)         ← 참조 데이터
interactions (댓글/좋아요/북마크)
search (통합 검색)              ← 4차에서 임베딩으로 swap
```

### 핵심 관계
- Story ↔ Category (FK)
- Comment.parent → self-FK (1단계 대댓글만)
- Precedent ↔ Law (M2M)
- Like / Bookmark: target_type + target_id (간단한 polymorphic)

### Soft Delete
- Story, Comment에 적용
- 기본 manager는 활성만 / `all_objects`는 admin/감사용
- 부모 deleted + 활성 reply가 있으면 placeholder로 트리 컨텍스트 보존

## 3. 기능 시연 (4분) — 시나리오 A

### 풀 시연 흐름
```
1. 회원가입 (demo_user) → 자동 로그인
2. 홈 검색창에 "전세 보증금을 못 받고 있어요" 입력
3. 검색 결과 페이지:
   - 추출 키워드: ["전세", "보증금"]  ← 조사 처리 동작 시연
   - 관련 법령: 주임법 7조 (차임 등의 증감청구권), 주임법 3조 (대항력)
   - 관련 판례: 임대차 보증금 반환 사건 (대법원, 원고일부승)
   - 비슷한 사연 5건
4. "주임법 7조" 클릭 → 법령 상세 + 관련 판례 + 북마크
5. 사연 "전세 보증금 반환 지연 문의" 클릭
   - 자동 검색 결과 (이 사연과 관련된 법령/판례/사연 자동 노출)
   - 댓글 트리 (testuser/testuser2 + 대댓글 + placeholder)
6. 댓글 작성 + 좋아요
7. 마이페이지 → 내 사연/북마크 확인
```

### 시연 포인트
- "조사 처리": `보증금을` → `보증금` 정확히 stripping
- "법령 먼저": 결과 페이지에서 법령이 가장 먼저 노출
- "자동 검색": 사연 상세 페이지가 그 사연 본문으로 자동 검색 → 페이지 진입만으로 관련 법/판례 노출
- "면책 고지": 모든 검색 결과에 disclaimer 필드 (법률 자문 X)

## 5. 한계점 + 4차 확장 (1분)

### 3차 한계점 (정직하게 짚기)
- **키워드 매칭만**: "보증금" 검색하면 보증금 단어가 들어간 것만 매칭. 의미적으로 비슷한 "임대차 보증" 등은 못 잡음.
- **법령/판례 데이터 8건만**: 구조 검증용. 본격 데이터는 STEP 8 이후.
- **단순 정렬**: 매칭 키워드 개수 기반. 가중치 X.
- **개인화 X**: 사용자 행동 기반 추천 X.

### 4차 확장 (이미 구조 잡혀 있음)
- **임베딩 검색**: `extract_keywords` → `generate_embedding` 으로 swap. 함수 시그니처 동일하게 설계됨. ViewSet/Serializer 수정 X.
- **외부 API**: 국가법령정보센터 OpenAPI 연동으로 본격 적재 (Celery 주기 작업)
- **FastAPI 분리**: 임베딩 모델 서빙 분리
- **Docker Compose**: Django + FastAPI + PostgreSQL(pgvector) + Celery + Redis + nginx 멀티 컨테이너
- **모니터링**: Prometheus + Grafana

### 마무리 한 줄
> "키워드 매칭이라는 단순한 도구로도 법령 → 판례 → 사연이 자연스럽게 묶이는 흐름을 만들었습니다. 4차에서 임베딩으로 갈아끼우면 같은 흐름이 더 똑똑하게 동작할 겁니다."

## Q&A 대비 (예상 질문)

| 질문 | 답변 |
|------|------|
| AI 답변 안 만드는 이유? | 변호사법 109조 회피. AI가 단정적 답변하면 법률 자문에 해당. 우리는 매칭만. |
| 케이스노트와 차이? | 케이스노트는 변호사용 빽빽한 UI. 우리는 일반인용 친근 톤. 자연어 입력. |
| 정확도? | 3차는 키워드 매칭이라 한계 명확. 4차에서 임베딩으로 정확도 개선. 함수 swap만으로. |
| 수익 모델? | 완전 무료 + 광고 + 정부 지원. 변호사 매칭/광고는 안 함 (위 변호사법 이유). |
| 시연 데이터 임시 마커? | `[임시] 2022다123456` 같은 case_number는 본격 적재 전 placeholder. README/SAMPLE_DATA에 명시. |
| Soft Delete 정책? | 사용자 실수 + 관리자 감사 위해 즉시 hard delete X. all_objects로 admin에서 복구 가능. |
| PostgreSQL 전환 이유? | 4차에서 pgvector로 임베딩 검색 갈 때를 위해 미리 PG로 전환. 한국어 ILIKE 동작 보장. |
